How to play

It's a turn-based battler. You build a party, pick a move each turn, and trade hits until one side faints. Move animations, status effects, and field hazards all factor in. There's a full story mode with saved progress, plus catch, PC, Underground, and Safari systems and boss arcs to work through.

One detail shapes every fight: HP and status fully restore between battles, but consumables only matter inside an encounter. So each fight is its own small puzzle: spend your items now, or save them for the turn that decides it. Play through the campaign against the AI, or take on another person in online PvP.

Under the hood

This is the project I've put the most into: a complete, stateful, networked game built and maintained solo, with design docs, database migrations, tests, and performance profiling behind it.

  • A custom engine in vanilla JavaScript. The turn loop, move resolution, and status/hazard system are all hand-written, with no game framework underneath, so I know exactly how every edge case resolves.
  • A Supabase / PostgreSQL backend with migrations handles the hard part: online PvP and saving progress. Two players acting on a shared, authoritative state is a different problem than a single-player loop, and most of the engineering effort lives here.
  • Performance profiling of the turn loop, move parsing, and memory, so the game stays responsive as the rule set grows.
  • A structured debug and QA system with a central issue ledger and spec-drift checks against the design docs: when the code and the design disagree, a check catches it instead of a player.
  • A PWA manifest so it installs and behaves like an app.

The test suite uses deterministic replay: the same seed reproduces a run exactly. That's the part I'm proudest of. A bug someone hits once becomes a test I can replay forever, so it never quietly comes back.

It's the same habit I bring to data work: make it reproducible, and measure it.
